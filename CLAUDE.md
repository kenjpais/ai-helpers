# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Claude Code plugins repository. Plugins live under `plugins/`.

## Plugin Structure

```text
plugins/{plugin-name}/
├── .claude-plugin/
│   └── plugin.json               # Required: name, description, version, author
├── commands/
│   └── {command-name}.md         # Required: at least one command
├── skills/                        # Optional
│   └── {skill-name}/
│       └── SKILL.md
└── README.md
```

Canonical example: `plugins/hello-world/`

## Development Commands

| Command | Purpose |
|---------|---------|
| `make help` | Show all available make targets |
| `make lint` | Run skillsaw validation (verbose, strict mode) — required before every commit |
| `make update` | Regenerate PLUGINS.md and docs/data.json after plugin changes |
| `make lint-pull` | Pull latest claudelint container image |

## Auto-Generated Files Workflow

When you modify plugins, several files must be regenerated and kept in sync:

1. **Edit plugin files** (commands, skills, plugin.json)
2. **Bump version** in `plugins/{name}/.claude-plugin/plugin.json` if needed (see Plugin Versioning)
3. **Run `make update`** which executes these scripts in sequence:
   - `fix_frontmatter_quotes.py` - Normalizes YAML frontmatter formatting
   - `sync_marketplace_versions.py` - Updates `.claude-plugin/marketplace.json` with versions from each plugin.json
   - `generate_plugin_docs.py` - Regenerates `PLUGINS.md` from all plugin metadata
   - `build-website.py` - Generates `docs/data.json` for the website

**Never edit these files manually**: PLUGINS.md, docs/data.json, .claude-plugin/marketplace.json

The `.skillsaw-custom.py` rule enforces this — lint will fail if these files are out of sync.

## Validation System

This repository uses [skillsaw](https://github.com/stbenjam/skillsaw) for plugin validation:

- **Configuration**: `.skillsaw.yaml` defines rule severity levels
- **Custom rules**: `.skillsaw-custom.py` implements repository-specific validation (e.g., ensuring PLUGINS.md stays in sync)
- **Auto-generated files**: PLUGINS.md and docs/data.json are regenerated by `make update` and must never be edited manually
- **CI enforcement**: GitHub Actions run lint checks and version validation on all PRs

## Plugin Versioning

All plugins use [semantic versioning](https://semver.org/):

- **PATCH** (0.0.x): Bug fixes, typos, minor improvements
- **MINOR** (0.x.0): New commands, skills, hooks, or features  
- **MAJOR** (x.0.0): Breaking changes to existing commands

**When to bump**: Modify `plugins/{name}/.claude-plugin/plugin.json` version for any code changes to commands, skills, hooks, or plugin.json itself. Documentation-only changes (README.md) do not require version bumps. CI will fail if you forget.

After bumping versions, run `make update` to sync `.claude-plugin/marketplace.json`.

## Contributing Rules

- **AI reasoning required.** Commands must require AI analysis/decisions, not just wrap scripts. If it could be a shell alias or Makefile target, it should not be a plugin command.
- **Never reference real people** by name, even stylistically. Describe desired qualities explicitly instead.
- **Check for overlaps** before contributing: run `/utils:review-ai-helpers-overlap --idea "description"`.
- **Follow existing patterns.** Read `plugins/hello-world/commands/echo.md` for command format; the linter enforces structure.
- **Use kebab-case** for all plugin names, command files, and skill directories.
- **Use `.work/{feature-name}/`** for temporary files (gitignored).
- **Register all plugins** in `.claude-plugin/marketplace.json`.
- **Set author** to `"github.com/openshift-eng"` in `plugin.json`.
- **Add new commands** to an existing plugin when they fit its scope, or to `plugins/utils/` if no clear parent. Create a new plugin only for a distinct group of related commands.

## Architecture

### Plugin Registry

- **marketplace.json**: Central registry at `.claude-plugin/marketplace.json` listing all plugins with versions
- **Auto-sync**: `scripts/sync_marketplace_versions.py` keeps marketplace.json in sync with individual plugin.json files
- **Website data**: `scripts/build-website.py` generates `docs/data.json` for https://openshift-eng.github.io/ai-helpers/

### Command Structure

Commands follow a standardized markdown format (enforced by skillsaw):

- **Frontmatter**: YAML with `description` and `argument-hint` fields
- **Sections**: Name, Synopsis, Description, Implementation, Return Value, Examples, Arguments
- **Pattern**: Inspired by Linux man pages (see `man 7 man-pages`)
- **Reference**: `plugins/hello-world/commands/echo.md`

## Go Code Guidelines

When writing Go code in this repository:

1. Follow [Effective Go](https://go.dev/doc/effective_go) principles
2. Run `gofmt -w .` before committing
3. Run `go vet` to catch common mistakes
4. Use idiomatic patterns: small interfaces, channels for communication, explicit error handling
5. Avoid anti-patterns: panic for normal errors, ignoring errors, storing context.Context in structs

## Python Code Guidelines

Python scripts in `scripts/` handle documentation generation and marketplace sync:

1. Scripts are Python 3 and should be run from repository root
2. No external dependencies required — uses stdlib only
3. Follow PEP 8 style conventions
4. Scripts are invoked by `make update` — test changes by running the full make target

## Testing Changes

- **Container-based testing**: Build and run the container with pre-installed plugins:
  ```bash
  podman build -f images/Dockerfile -t ai-helpers .
  podman run -it -v $(pwd):/workspace -w /workspace ai-helpers
  ```
- **Local validation**: Always run `make lint` before committing
- **CI checks**: GitHub Actions validate plugin versions, run linter, and check docs are up-to-date
